# In-Memory Store Model and `resetStores` Test Contract

This document describes every mutable store exported from [`src/stores.ts`](../src/stores.ts), the lifecycle of that state, and the test contract enforced by `resetStores()`.

---

## Overview

All mutable runtime state for the StableRoute backend lives in `src/stores.ts`. There is no database adapter yet; every store is a process-local JavaScript data structure. **State is lost on process restart.** Persistence via a database adapter is planned but not yet implemented.

Centralising state in one module lets:

- Route handlers in `src/index.ts` import and mutate stores through typed accessors.
- The test suite import `resetStores` and call it between cases to prevent cross-test bleed.

---

## Stores

### `pairRegistry` — `Set<string>`

Holds the canonical key for every registered trading pair.

| Property | Value |
|----------|-------|
| Type | `Set<string>` |
| Key format | `"SOURCE::DEST"` — see [`pairKey`](#pairkey) below |
| On-chain mirror | `DataKey::Pair(source, dest)` |
| Initial value | empty `Set` |

**Example entries:** `"USDC::EURC"`, `"EURC::USDC"`.

---

### `pairMeta` — `Map<string, PairMeta>`

Per-pair fee and liquidity metadata keyed by the same `pairKey` string used in `pairRegistry`.

```typescript
type PairMeta = {
  feeBps: number;      // fee in basis points (100 = 1 %)
  minAmount: string;   // minimum swap amount (string to avoid float precision loss)
  maxAmount: string;   // maximum swap amount
  liquidity: string;   // available liquidity
};
```

| Property | Value |
|----------|-------|
| Type | `Map<string, PairMeta>` |
| On-chain mirror | `DataKey::PairFeeBps / Min / Max / Liquidity` |
| Initial value | empty `Map`; new entries start from `defaultMeta()` |

---

### `apiKeyStore` — `Map<string, ApiKeyRecord>`

API keys generated by the backend, keyed by the key's non-secret 8-char
`prefix` — never by the raw key itself. Each record stores only a salt and
a keyed hash of the raw key (never the raw key), so a leaked snapshot never
exposes usable credentials. See [api-key-auth.md](api-key-auth.md) for the
full hashing/verification/migration story.

```typescript
type ApiKeyRecord = {
  label: string;
  createdAt: number; // Unix timestamp (ms)
  scopes?: string[];
  rotatedAt?: number;
  graceExpiresAt?: number;
  expiresAt?: number;
  lastUsedAt?: number;
  salt: string;   // hex-encoded random salt
  hash: string;   // HMAC-SHA256(salt, rawKey), hex-encoded
};
```

---

### `webhookStore` — `Map<string, WebhookRecord>`

Registered webhooks keyed by a UUID webhook id.

```typescript
type WebhookRecord = {
  url: string;
  events: string[];
  createdAt: number; // Unix timestamp (ms)
};
```

---

### `eventLog` — `AppEvent[]`

Bounded ring-buffer of application events. Capped at `EVENT_LOG_CAP` (10 000 entries); the oldest entry is evicted when the cap is exceeded.

```typescript
type AppEvent = {
  id: string;                        // UUID
  ts: number;                        // Unix timestamp (ms)
  type: string;
  payload: Record<string, unknown>;
};
```

Use `recordEvent(type, payload)` to append; **never push directly** so the cap is respected.

---

### `rateBuckets` — `Map<string, number[]>`

Per-IP sliding-window timestamps used by the rate limiter. Each key is a client IP address; the value is a list of request timestamps (ms) within the current window.

---

### `config` — `Record<string, number>`

Runtime configuration tunable via `GET /api/v1/config` and `PATCH /api/v1/config`.

| Key | Default | Description |
|-----|---------|-------------|
| `rateLimitPerWindow` | `60` | Max requests per window per IP |
| `rateLimitWindowMs` | `60000` | Window size in milliseconds |
| `bulkMaxItems` | `100` | Max items accepted by bulk endpoints |
| `eventLogCap` | `10000` | Mirror of `EVENT_LOG_CAP` |

---

### `paused` — `boolean`

Service-level pause flag. When `true`, the pause-guard middleware rejects all non-idempotent requests with HTTP `503`. Mutate through `setPaused(value)` — do not reassign the binding directly.

---

## Helper Functions

### `pairKey(source, dest)`

Derives the canonical pair key from two asset codes.

```typescript
pairKey("USDC", "EURC") // → "USDC::EURC"
```

The `::` separator is chosen to be URL-safe and unlikely to appear in asset codes. All lookups against `pairRegistry` and `pairMeta` must use this function to produce consistent keys.

### `defaultMeta()`

Returns a fresh `PairMeta` with all fields zeroed:

```typescript
{ feeBps: 0, minAmount: "0", maxAmount: "0", liquidity: "0" }
```

Use this as the baseline when registering a new pair before its on-chain data has been synced.

## Lifecycle

```
Process start
  ├─ Module initialised → all stores empty / at defaults
  └─ If PERSIST_PATH is set → JsonFileStoreAdapter loads snapshot from disk
       ├─ File exists & valid → hydrate stores from snapshot
       └─ File missing / corrupt / wrong shape → fall back to empty stores
           (console.warn / console.error emitted)

Runtime
  ├─ Route handlers read and mutate stores directly
  └─ Mutations trigger debounced auto-save (100 ms debounce)
       ├─ writeFileSync + renameSync → snapshot written atomically
       └─ I/O failure → error logged, in-memory state unaffected

Process exit / crash / restart
  └─ Latest snapshot read from disk on next start (if PERSIST_PATH is set)
```

There is no lazy-initialisation; state is ready the moment `src/stores.ts` is first imported.

## Persistence (Snapshot to Disk)

The `PERSIST_PATH` environment variable controls whether store state is persisted
to disk as a JSON snapshot between restarts.

### Adapter selection

| `PERSIST_PATH`            | Adapter used                    | Behaviour                              |
|---------------------------|---------------------------------|----------------------------------------|
| unset / empty / `""`      | `InMemoryStoreAdapter`          | No disk I/O; state lost on restart     |
| set to a file path        | `JsonFileStoreAdapter`          | State snapshotted to JSON file on disk |

The factory function `getStoreAdapter()` (from `src/persistence.ts`) checks
`process.env.PERSIST_PATH` on every call and returns the matching adapter.

```typescript
import { getStoreAdapter } from "./persistence";
const adapter = getStoreAdapter();
```

### Snapshot shape (`StoreSnapshot`)

```typescript
interface StoreSnapshot {
  pairRegistry: string[];
  pairMeta: [string, PairMeta][];
  apiKeyStore: [string, ApiKeyRecord][];
  webhookStore: [string, WebhookRecord][];
  eventLog: AppEvent[];
}
```

### Recovery from corrupted snapshots

`JsonFileStoreAdapter.load()` returns `null` (triggering a clean empty-store
fallback) when the snapshot file:

- Does not exist.
- Is empty.
- Contains invalid (unparseable) JSON.
- Contains valid JSON that does not match the `StoreSnapshot` shape — e.g. an
  array, a plain object with wrong field names, or a truncated object.

A console warning is emitted on shape mismatches; parse errors are logged via
`console.error`.  The process **never** crashes due to a corrupted snapshot
— it falls back to fresh empty stores.

### I/O failure resilience

`JsonFileStoreAdapter.save()` writes atomically via a temp-file-then-rename
strategy.  If `writeFileSync` or `renameSync` throws (e.g. `EACCES` when the
target directory is unwritable):

1. The error is logged via `console.error`.
2. Any leftover `.tmp` file is cleaned up (best-effort, failures during
   cleanup are silently ignored).
3. The in-memory store state is **never** affected — the process continues
   serving from memory.

This means the backend is resilient to transient filesystem issues and
misconfigured snapshot paths.  The debounced auto-save mechanism in
`triggerSnapshot()` catches any adapter rejection and logs it, preventing
unhandled promise rejections.

## `resetStores` Test Contract

```typescript
export const resetStores = (): void => { /* ... */ };
```

`resetStores` clears every store back to its initial state:

- `pairRegistry.clear()`
- `pairMeta.clear()`
- `apiKeyStore.clear()`
- `webhookStore.clear()`
- `eventLog.length = 0`
- `rateBuckets.clear()`
- `config` re-populated from `defaultConfig()`
- `paused` reset to `false`

### Rules for test authors

1. **Call `resetStores` in `beforeEach` (or `afterEach`)** for every test suite that touches any store.

   ```typescript
   import { resetStores } from "../src/stores";

   beforeEach(() => {
     resetStores();
   });
   ```

2. **Never call `resetStores` from production code.** It is not reachable via any HTTP route and must stay that way. Adding an HTTP route that calls it would expose a critical denial-of-service vector.

3. **Omitting `resetStores` causes cross-test bleed.** Data written by one test case leaks into subsequent cases, leading to non-deterministic failures that are hard to reproduce in isolation.

4. **`resetStores` is synchronous.** It does not need to be awaited.

### Why a single module?

Keeping all stores in one file makes the reset trivial: a single function call is the only entry point tests need. If stores were scattered across multiple modules, each module would need its own reset export, and test setup would become fragile.

---

## Security Notes

- `resetStores` and the raw store references (`pairRegistry`, `pairMeta`, etc.) are **test-only exports**. They must never be imported in HTTP route handlers or exposed over the network.
- `config` is the only store intentionally exposed via HTTP (`GET/PATCH /api/v1/config`). All other stores are internal.
